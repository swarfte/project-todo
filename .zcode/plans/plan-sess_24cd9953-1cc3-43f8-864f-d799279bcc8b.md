## Step Page Implementation Plan

Implementing a linear step chain per task, mirroring the task page's patterns. Per your answers: steps follow the `previousStepId` chain, read-only numbered list (no reorder UI yet), and tapping a task's title navigates into its steps.

### File 1: `lib/api.dart` — fix the stub + add count

- **Fix `getStepListByTaskId`**: change return type from `Future<bool>` → `Future<List<RecordModel>>`, return the actual records. Mirrors `getTaskListByProjectId`.
- **`createStep`**: already sets `isCompleted: false` and accepts `previousStepId`. New steps created from the FAB will be appended (callers pass the current last step's id). No change to signature.

### File 2: `lib/components/chainTimeline.dart` — add tap-to-open

- Add a new required callback `onOpen` (`void Function(Task task)`) to both `ChainTimeline` and `TreeRow`, threaded through like the existing `onEdit`/`onDelete`.
- Wrap the task title/content `Expanded` block in an `InkWell`/`GestureDetector` with `onTap: () => onOpen(task)`. The gutter badge's existing completion toggle is untouched. Add a subtle chevron/affordance hint so the tap target is discoverable.

### File 3: `lib/pages/task.dart` — wire navigation

- Add `_openStepPage(Task task)` that pushes a `MaterialPageRoute` to `StepPage(task: task)` via `await Navigator.push`. After it returns, call `_loadTasks()` (a step's completion may affect the task's apparent progress and is cheap to refresh — guarded by `mounted`).
- Wire `onOpen: _openStepPage` into the `ChainTimeline`.
- Import `step.dart`.

### File 4: `lib/pages/step.dart` — the new page (rewrite the stub)

Mirrors `task.dart`'s structure but linear. Constructor takes `final Task task`. AppBar title = `task.name`. Color scheme: amber[600] to visually distinguish it from teal task page / blue project page.

State:
- `_steps`, `_isLoading`, `_loadError`, `final APIService _apiService`.

Methods (mirror `task.dart` exactly):
- `_loadSteps()`: set loading → `getStepListByTaskId(task.id)` → map through `StepAdaptor.fromJson` → `setState`. Try/catch like `_loadTasks`.
- `_orderSteps(List<Step>)`: builds the linear chain from `previousStepId`. Find head (no predecessor / predecessor not in this task's set / self-loop → head). Walk forward following `previousStepId` inverse map. Cycle guard + safety net (append unreached). Returns the ordered `List<Step>`.
- `_openCreateStepDialog()`: FAB handler. If steps exist, pass the last step as `previousStepId` (append). Refresh on close.
- `_openEditStepDialog(Step step)`: opens `EditStepDialog`. Refresh on close.
- `_confirmDeleteStep(Step step)`: confirm `AlertDialog` → `deleteStep` → snackbar + reload. Mirror `_confirmDeleteTask`.
- `_toggleComplete(Step step)`: build updated `Step`, `updateStep`, reload. Mirror task's `_toggleComplete` (no completedAt field on Step, just flip isCompleted).

`build`: loading spinner / error+retry / empty-state ("No steps yet. Tap + to add the first step.") / else `ListView` of `_StepListTile`s inside a `RefreshIndicator`. FAB bottom-right, `onPressed: _openCreateStepDialog`.

`_StepListTile` (private widget in same file): a `Card`/`ListTile` showing:
- Leading: a numbered circle badge (1, 2, 3...) or a checkbox-style circle that toggles completion on tap (mirror the task badge: blue when pending, green check when done).
- A vertical connector line between steps drawn via a simple `Container`-border or `CustomPaint` to show "follow one by one" linear chain — lightweight, not the elaborate tree painter.
- Title (strike-through + grey when completed), subtitle = step number / "Step N of M".
- Trailing: a `PopupMenuButton` with Edit / Delete (same style as the task row menu).

### File 5: `lib/components/createStepDialog.dart` — NEW

Shows a hint: "Added as step N" or "Comes after '<prevName>'" if `previousStepId` set. Validation + `_isSending` guard + `SuccessSnackBar` + pop on success — same skeleton as `createTaskDialog`.

### File 6: `lib/components/editStepDialog.dart` — NEW

Slimmed `editTaskDialog`. Constructor: `({required Step step})`. Fields: name `TextField`, a completed checkbox/switch. No previous-step dropdown (no reorder this pass — read-only chain). Saves via `updateStep`. Same dialog skeleton (error box, LinearProgressIndicator, Cancel/Save).
